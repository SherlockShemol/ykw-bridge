# 4.3 フェイルオーバー

## 概要

フェイルオーバーは、現在の provider へのリクエストが失敗したときに、キュー内の次の provider へ自動で切り替える仕組みです。

フェイルオーバーはアプリごとに設定され、現在対応する対象は:

- Claude Code
- Claude Desktop

## 前提条件

実際にフェイルオーバーを機能させるには、次のすべてが必要です:

1. ルーティングサービスが起動している
2. 対象アプリで takeover が有効
3. フェイルオーバーキューが設定されている
4. 自動フェイルオーバーが有効

## キュー設定

フェイルオーバーパネルで:

1. 設定したいアプリを選ぶ
2. 利用可能な provider をキューへ追加する
3. 並び順で優先順位を決める

順序ルール:

- 前にあるほど優先度が高い
- 現在のリクエストが失敗したら後続 provider を順に試す

## クイック操作

メイン画面のフェイルオーバートグルを有効にしている場合、provider カードから直接:

- キューへ追加
- キューから削除

ができます。

## 自動フェイルオーバー

自動フェイルオーバーが無効な場合:

- 失敗は記録される
- provider は自動切替されない

自動フェイルオーバーが有効な場合:

- 失敗した provider の代わりにキュー内の次の provider へ自動切替される

## サーキットブレーカー

YKW Bridge は失敗が連続する provider に対して circuit breaker 保護をかけ、明らかに不健康なノードへ流量を打ち続けないようにします。

主な状態:

- **Closed**: 正常で利用可能
- **Open**: 一時的に遮断されスキップされる
- **Half-Open**: 回復確認中

主な調整項目:

- 最大リトライ回数
- first byte timeout
- stream idle timeout
- 連続失敗しきい値
- 回復待機時間
- エラー率しきい値

Claude Code と Claude Desktop は現在同じ Claude-like デフォルト調整値を共有します。

## 健康状態

キューやカードには健康状態が表示されます:

- 緑: healthy
- 黄: degraded / 一部失敗あり
- 赤: tripped

## 推奨

- 最初には最も安定した主力 provider を置く
- 2 番目には速度と安定性のバランスが良い代替を置く
- 最後には遅くても安定した保険を置く

## よくある問題

### なぜ自動切替されなかったのですか？

次を順に確認してください:

- ルーティングサービスは動いているか
- アプリは takeover 済みか
- 自動フェイルオーバーは有効か
- キューに本当に代替 provider があるか

### なぜ頻繁に切り替わるのですか？

よくある原因:

- 主 provider が不安定
- breaker しきい値が敏感すぎる

しきい値を少し緩めるか、主 provider を入れ替えてください。
