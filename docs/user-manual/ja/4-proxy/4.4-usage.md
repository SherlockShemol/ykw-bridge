# 4.4 使用量統計

## 概要

使用量統計は、リクエストログ、token 消費量、モデルコストを集計し、次を観察できるようにします:

- リクエスト量
- 成功率
- token 使用量
- 推定コスト
- provider / モデル分布

現在の主なデータソースは 2 つです:

| データソース | 対象範囲 | プロキシ必須か |
| --- | --- | --- |
| ルーティングリクエストログ | YKW Bridge を通過したすべてのリクエスト | はい |
| Claude セッションログ | ローカル Claude セッション内の使用量情報 | いいえ |

## アプリフィルター

使用量ページのフィルターは現在次の通りです:

- All
- Claude
- Claude Desktop

> ここでの "Claude Desktop" は主に Claude Desktop takeover チェーン経由で生成されたプロキシログを区別するためのものです。セッションログ同期は現在 Claude セッションディレクトリのみを対象にします。

## ページを開く

**設定 → Usage** を開きます。

## ページ内容

### サマリーカード

上部に次が表示されます:

- 総リクエスト数
- 総 token
- 推定コスト
- 成功率

### トレンドチャート

次の期間で傾向を確認できます:

- 24 時間
- 7 日
- 30 日

### 詳細テーブル

下部には 3 種類の統計があります:

- リクエストログ
- provider 統計
- モデル統計

## リクエストログ

各ログ項目には次が表示されます:

- 時刻
- アプリ種別
- Provider
- モデル
- 入力 / 出力 token
- cache read / cache creation
- 推定コスト
- レイテンシ / time to first byte
- HTTP ステータス

## モデル料金

高度設定でモデルごとのカスタム料金を管理できます。

主な項目:

- モデル ID
- 表示名
- 入力価格
- 出力価格
- cache read 価格
- cache creation 価格

YKW Bridge はまずリクエスト中のモデル名を正規化し、その後で料金表と照合します。

## `codex` / `gemini` を含むモデル名について

モデル統計や料金設定に `gpt-*-codex` や `gemini-*` が見える場合、それは:

- **provider が実際に返しているモデル名**
- YKW Bridge が古い standalone app 入口をまだ公開しているという意味ではない

現在の版でも、Claude ワークフローは互換 provider を通じてそれらのモデル名を利用できます。

## セッションログ同期

**Sync Session Logs** を押すと、YKW Bridge はローカル Claude セッションを走査し、解析可能な使用量記録を取り込みます。

この処理にプロキシは不要ですが、ローカル Claude セッションファイルが存在している必要があります。

## よくある問題

### なぜデータが表示されないのですか？

まず確認すること:

- プロキシがリクエストを記録しているか
- 実際のリクエストが通っているか
- Claude セッションディレクトリに解析可能なログがあるか

### なぜコストが公式請求と完全一致しないのですか？

よくある理由:

- 一部モデルにまだ正確な料金設定が入っていない
- provider 側で独自倍率が使われている
- 公式請求とローカル推定で丸め方が異なる
