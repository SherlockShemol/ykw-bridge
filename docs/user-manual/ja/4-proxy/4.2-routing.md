# 4.2 アプリ takeover

## 概要

アプリ takeover とは、クライアントがまずローカルの YKW Bridge ルーティングサービスへリクエストを送り、その後どの provider が実際に処理するかを YKW Bridge が決める仕組みです。

現在対応する対象:

- `claude`
- `claude_desktop`

古いワークフロー向けの takeover フローはもう存在しません。

## 前提条件

アプリ takeover を機能させるには、先にルーティングサービスが起動している必要があります。

## 有効化場所

**設定 → ルーティングサービス** で次を個別に切り替えられます:

- Claude Code takeover
- Claude Desktop takeover

## takeover 後の挙動

### Claude Code

YKW Bridge は Claude Code のリクエスト先をローカルプロキシアドレスへ書き換えます。

これにより:

- リクエストを記録できる
- フェイルオーバーに参加できる
- provider 切替が即時反映される

### Claude Desktop

YKW Bridge は managed gateway 設定を書き込み、ローカル HTTPS と gateway secret を通して takeover を完了します。

これにより次も有効になります:

- リクエストログ
- フェイルオーバー
- ホットスイッチ
- Doctor、証明書、起動チェーンとの連携

## takeover 中の切替

ルーティングサービスが動作中で、アプリがすでに takeover 済みの場合:

1. メイン UI で provider を切り替える
2. プロキシ層が直ちに新しい provider へ新規リクエストを転送する
3. **クライアント再起動は不要**

これがホットスイッチの中心的な価値です。

## takeover を無効にする

takeover を無効にすると、YKW Bridge は以前の live 設定またはバックアップ内容の復元を試みます。

復元されない場合の主な原因:

- 途中で異常終了した
- 別のツールが live ファイルを書き換えた

最も安全な回復手順:

1. 現在の provider を開く
2. もう一度保存する
3. または takeover を再度有効にしてから正常に無効化する

## 注意事項

- takeover はローカルのルーティングサービスに依存します。サービスが動いていないとトグルは実質的に何もしません
- Claude Desktop の takeover は Claude Code より複雑なので、初回利用前に Doctor を実行するのがおすすめです
- サードパーティ reverse proxy や managed OAuth 経路で公式 API を使う場合は、自分でリスクを評価してください
