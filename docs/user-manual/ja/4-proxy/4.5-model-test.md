# 4.5 モデルテスト (Stream Check)

## 概要

モデルテストは現在の provider に実際のテストリクエストを送り、次を確認します:

- モデルが存在するか
- 認証情報が有効か
- エンドポイントへ到達できるか
- レイテンシが許容範囲か
- ストリームの first token までの時間が妥当か

現在の Stream Check は **Claude-like provider** のみを対象とし、Claude Code と Claude Desktop は同じテスト設定モデルを共有します。

## 設定

**設定 → Advanced → Model Test Configuration** で次を調整できます:

| 設定 | 説明 | デフォルト |
| --- | --- | --- |
| Claude Test Model | テストに使うモデル名 | システムデフォルト |
| Timeout | 単一テストリクエストの制限時間 | 45 秒 |
| Max Retries | 失敗後の再試行回数 | 2 |
| Degrade Threshold | これを超えると degraded 扱い | 6000ms |
| Test Prompt | モデルへ送る短いプロンプト | `Who are you?` |

## 推奨

- 安くて速くて安定したテストモデルを選ぶ
- Claude ワークフローでは Haiku クラスの小さいモデルが扱いやすい
- テストプロンプトは短く保つ

## 手動テスト

provider カードで **Test** を押すと、YKW Bridge は:

1. 最小限のリクエストを構築
2. 現在の provider へ送信
3. 総応答時間と first byte を計測
4. 健康状態を返す

## 結果状態

主な結果:

- **Healthy**: しきい値内で成功
- **Degraded**: 成功したがしきい値より遅い
- **Unavailable**: 失敗、タイムアウト、またはモデル利用不可

## フェイルオーバーとの関係

ルーティングサービス有効時、Stream Check の結果はフェイルオーバーや健康判定に利用されます:

- 健康なノードは優先される
- 失敗が続くノードは circuit breaker に入ることがある
- 回復中ノードは再プローブされる

## 注意事項

- モデルテストでは少量の quota を消費します
- テストモデル自体の指定が誤っていると、「テスト失敗だが実運用は動く」状態になることがあります
- Claude Code と Claude Desktop は現在同じテストモデル設定を共有します
