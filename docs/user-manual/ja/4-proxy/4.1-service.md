# 4.1 ルーティングサービス

## 概要

ルーティングサービスはローカルの HTTP/HTTPS プロキシを起動し、リクエスト転送、ログ記録、フェイルオーバー、そして takeover 有効時のホットスイッチを提供します。

現在の対象:

- **Claude Code**
- **Claude Desktop**

## 起動方法

次の 2 つの入口から起動できます:

1. メイン画面上部のルーティングトグル
2. 設定内の **ルーティングサービス** パネル

## 基本設定

主な設定項目:

| 設定 | 説明 | デフォルト |
| --- | --- | --- |
| Listen Address | プロキシがバインドするアドレス | `127.0.0.1` |
| Listen Port | プロキシポート | `15721` |
| Enable Logging | リクエストログを記録するか | 有効 |

Listen Address や Port を変更したい場合は、先にサービスを停止してから保存し、その後再起動するのが安全です。

## 実行中情報

起動後、パネルには次が表示されます:

- 現在の listen address
- アクティブ接続数
- 総リクエスト数
- 成功率
- 稼働時間
- 現在転送先となっている provider
- 各アプリのフェイルオーバーキュー

現在の provider はアプリごとに分けて表示されます。例:

- Claude Code
- Claude Desktop

## リクエストの流れ

典型的な流れ:

1. Claude Code または Claude Desktop がリクエストを送る
2. YKW Bridge がリクエストログと使用量を記録する
3. 現在の provider とフェイルオーバー戦略に従って転送する
4. クライアントにレスポンスを返す

## プロキシ takeover

ルーティングサービス起動後、特定アプリに対して **takeover** を有効化できます。

### Claude Code

YKW Bridge は Claude Code のリクエスト先をローカルプロキシアドレスへ書き換えます。

### Claude Desktop

YKW Bridge は Claude Desktop 用の managed gateway 設定を書き込み、ローカル証明書とローカル gateway を通して転送します。

> Claude Desktop の takeover チェーンには、証明書、gateway secret、launch shim、watchdog などが含まれるため、単純な base URL 差し替え以上の処理になります。

## API 形式変換

Claude ワークフロー内で provider がネイティブな Anthropic Messages 形式を使わない場合、プロキシ層がリクエスト / レスポンス形式を自動変換します。

現在主に対応する形式:

- Anthropic Messages: そのまま通す
- OpenAI Chat Completions: 双方向変換
- OpenAI Responses: 双方向変換

これにより、Claude ワークフローでより多くのサードパーティ provider を使えるようになります。

## サービス停止

ルーティングサービス停止時、YKW Bridge は次を試みます:

1. live 設定を復元する
2. バックアップを保持する
3. 現在の listener を閉じる

takeover を有効にしている場合は、強制終了よりも通常の停止を使うほうが安全です。

## ログと確認場所

リクエストログは **設定 → Usage** に表示され、次の用途に使えます:

- リクエスト詳細の確認
- コスト見積もり
- 失敗したリクエストの調査
- フェイルオーバー挙動の観察

## よくある問題

### ポートが使用中

アドレスがすでに使用中の場合:

1. 別のポートを使う
2. そのポートを使っているプロセスを停止する

### サービスが起動しない

まず確認すること:

- listen address と port が正しいか
- 証明書や権限状態に問題がないか（特に Claude Desktop）
- 古いインスタンスが残っていないか
