# 4.2 App Takeover

## Overview

App takeover means a client sends its requests to the local YKW Bridge routing service first, and YKW Bridge then decides which provider should actually handle them.

Currently supported:

- `claude`
- `claude_desktop`

Legacy takeover flows for older workflows no longer exist.

## Prerequisite

The routing service must already be running before app takeover can do anything.

## Where to Enable It

In **Settings → Routing Service**, you can separately control:

- Claude Code takeover
- Claude Desktop takeover

## Behavior After Takeover

### Claude Code

YKW Bridge rewrites Claude Code requests to the local proxy address.

Benefits:

- Requests are logged
- Failover can participate
- Provider switching can take effect immediately

### Claude Desktop

YKW Bridge writes a managed gateway configuration and completes takeover through local HTTPS and gateway secret handling.

That also enables:

- Request logging
- Failover
- Hot switching
- Doctor, certificate, and launch-chain integration

## Switching While Takeover Is Active

When the routing service is running and the app is already taken over:

1. Switch the provider in the main UI
2. The proxy layer immediately begins forwarding new requests to the new provider
3. **No client restart is required**

That is the core value of hot switching.

## Turning Takeover Off

When takeover is disabled, YKW Bridge attempts to restore the previous live config or backup content.

If you find that restore did not happen, the usual reasons are:

- An abnormal exit happened midway
- Another tool rewrote the live file

The safest recovery path is:

1. Open the current provider
2. Save it again
3. Or re-enable takeover and then disable it cleanly

## Notes

- Takeover depends on the local routing service; if the service is not running, the toggle has no practical effect
- Claude Desktop takeover is more complex than Claude Code takeover; it is a good idea to run Doctor before first use
- If you access official APIs through third-party reverse proxies or managed OAuth paths, evaluate the risk yourself
