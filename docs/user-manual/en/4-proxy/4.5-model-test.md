# 4.5 Model Test (Stream Check)

## Overview

Model Test sends a real test request to the current provider to verify:

- Whether the model exists
- Whether credentials are valid
- Whether the endpoint is reachable
- Whether latency is acceptable
- Whether time to first streamed token is reasonable

The current Stream Check serves only **Claude-like providers**, which means Claude Code and Claude Desktop share the same test configuration model.

## Configuration

In **Settings → Advanced → Model Test Configuration**, you can adjust:

| Setting | Description | Default |
| --- | --- | --- |
| Claude Test Model | Model name used for testing | System default |
| Timeout | Timeout for a single test request | 45 seconds |
| Max Retries | Retry count after failure | 2 |
| Degrade Threshold | Mark as degraded when latency exceeds this | 6000ms |
| Test Prompt | Short prompt sent to the model | `Who are you?` |

## Recommendations

- Prefer a cheap, fast, stable test model
- For Claude workflows, a smaller model such as a Haiku-class model is usually a good choice
- Keep the test prompt short

## Manual Test

When you click **Test** on a provider card, YKW Bridge:

1. Builds a minimal request
2. Sends it to the current provider
3. Measures total response time and time to first byte
4. Returns a health status

## Result States

Common results:

- **Healthy**: request succeeds within the threshold
- **Degraded**: request succeeds but is slower than the threshold
- **Unavailable**: request fails, times out, or the model is unavailable

## Relationship to Failover

When the routing service is enabled, Stream Check results contribute to failover and health decisions:

- Healthy nodes are preferred
- Repeatedly failing nodes may trip the circuit breaker
- Recovering nodes are probed again

## Notes

- Model tests consume a small amount of quota
- If the test model itself is misconfigured, you can get "test failed but real usage still works"
- Claude Code and Claude Desktop currently share the same test model configuration
